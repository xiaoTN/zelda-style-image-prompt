# 塞尔达游戏实况截图视觉风格

本文档是 `zelda-style-image-prompt` 的内置视觉规范。写提示词前先按本文档确定画风、角色、HUD、中文 UI、区域发现地名 UI 和元素池。

## 1. 画面总目标

默认按"原版塞尔达游戏实况截图复刻"方向写 prompt，不是普通奇幻插画、不是电影 CG、不是动漫风、不是现代 App 截图。

- **渲染风格描述（用专门的 `画风` 字段集中锚定，正反双向）**：prompt 必须有一个**单独的 `画风` 字段**，集中写原版实机截图风格和反向声明禁词，把模型从写实/CG 方向拉回 BOTW/TOTK 实机心锚。**核心心锚词 + 画风字段是绑定的——心锚词单写一行（如 `塞尔达原版游戏实况截图复刻`），紧接着一行 `画风：...` 字段。**

  **画风字段标准写法**（直接复制使用）：

  ```text
  塞尔达原版游戏实况截图复刻
  画风：任天堂 Switch 旷野之息/王国之泪实机游戏截图风格；不要电影 CG、不要写实摄影、不要 HDR 真实光影、不要湿润真实材质
  ```

  - 画风字段只保留**原版实机游戏截图风格 + 反向声明禁词**——不要电影 CG / 不要写实摄影 / 不要 HDR / 不要湿润真实材质 等写在这一行里；不要再追加 `cel-shading 扁平卡通色块、强黑描边轮廓、柔光绘画感、淡雅手绘风色调` 这类风格串
  - 其他字段（地点 / 画面要素 / 玩法时刻 / 塞尔达元素 / HUD / 文字）**不要重复写**这些渲染词，也不要在场景描述里夹带"真实/写实/质感/材质/光影"等引导写实的副词
  - 历史背景：曾尝试只写一句心锚词、把技术细节完全交给模型自处理，实测在含真实景区题材（土黄色废墟、藏式建筑、雨林）下模型容易滑向 HDR/电影 CG/写实摄影质感；后改为「单独画风字段集中锚定 + 反向声明禁词」，渲染卡通感稳定可控
  - **严禁**把画风技术词散写到多个字段、或在场景描述里夹带"真实景区"、"真实地理特征"、"真实建筑"、"地理质感"等引导词；画风字段以外的字段写法仍保持极简，元素只列名称、不展开外观
  - **画风版实测结构**：单张场景图优先使用 `心锚词 -> 画风 -> 游戏版本/角色 -> 海拉鲁地图区块 -> 画面要素 -> 玩法时刻 -> 塞尔达元素 -> 地名 UI 文字（区域发现） -> HUD -> 文字` 的 10-13 行结构。地点特征压成 3-5 个名词短语，塞尔达元素只列 3-7 个名称关键词，人物动作只写一句玩法时刻并标注是否消耗体力。不要回退到 15 行以上的长分镜、前中远景分层、材质颜色光线堆叠或写实景区字段名。

  本文档下面列出的"严禁写实摄影质感、电影 CG 光线追踪、湿润真实材质、HDR 真实光影"等条款，是给 **prompt 自检和人工审图时的判断依据**，也是 prompt 画风字段反向声明的取词来源。
- **相机**：第三人称游戏镜头，林克在前景或中景背影/侧后方，像玩家正在操作角色探索。`16:9 横图` 由出图 wrapper 的尺寸参数指定，prompt 字段里**不需要再写**画面比例。
- **角色造型（不要写进 prompt）**：BOTW 和 TOTK 各只有一种林克 / 塞尔达造型，模型从"游戏版本心锚"自处理。**prompt 里不要展开**林克 / 塞尔达的衣服颜色、披风颜色、袖口镶边、护腕、靴子、头发颜色 / 长度 / 马尾 / 编辫位置、耳坠颜色、眼瞳颜色、背挂武器具体清单（大师剑 / 哈特诺剑 / 希卡蓝盾 / 海拉鲁圆盾 / 海拉鲁长弓）等任何造型细节——这些都是 BOTW/TOTK 各自版本的默认值，模型从"塞尔达原版游戏实况截图复刻"+ 角色名（"林克" / "塞尔达"）+ 可选游戏版本（BOTW / TOTK）就能完整复现。
  - 在 `林克造型` / `塞尔达造型` 字段里只要写一个版本号或"BOTW/TOTK 默认造型"就够。攻略图整组复用同一套装备款式时，只在第一段写一次"本组按 TOTK 默认造型"作为版本心锚即可，不要每张 prompt 整段复述装备清单。
  - 表情按原版"冷静专注或淡淡微笑"心锚，不需要在每张 prompt 字段里强调，模型默认就生成对的。
  - 本文档下面列出的具体造型细节（披风颜色、袖口镶边、希卡蓝盾等）是给**人工审图判断**用的——人工审图时如果发现造型偏离 BOTW/TOTK 默认值就按用户反馈决定是否重新生成，但 prompt 里不需要主动展开。
  - **真正需要在 prompt 里强调的反例**：用户明确要求"林克穿格鲁德女装" / "林克在沙漠改穿沙漠装" / "塞尔达穿研究员变装" 等**非默认造型**时，才需要在 prompt 里展开一句"林克本图穿沙漠白衫 + 头巾"。否则不写。
- **NPC 现代化禁忌仍生效**：上面"角色造型不要写进 prompt"只适用于林克 / 塞尔达 / 标准海拉鲁种族 NPC；真实景区里的游客 / 工作人员 / 路人**仍然要主动 NPC 化**——prompt 里写"游客转译为海拉鲁村民 / 驿站老板 / 旅行商人 / 希卡族 NPC 等"，避免模型把游客画成现代游客或现代民族服装。
- **动作姿态（只写动词，不写身体微观细节）**：动作字段只写一个动作动词或极短动宾词组——`在攀爬` / `举平板调查` / `骑马加速` / `张开滑翔伞` / `持剑格挡` / `烤蘑菇`。**严禁**写身体微观细节：双手刚抓住把手 / 伞才半展开 60% / 伞面斜上方 / 双膝微屈蓄力 / 脚仍未离地 / 起跳前 0.5 秒 / 最后一帧 / 过渡动画中间帧 / 左手扣岩点 / 右手反握 / 重心偏移 / 披风受力 / 瞳孔朝向 / 头部转向角度 / 肌肉发力 / 身体倾斜 等。也不要写抽象技能名（`使用攀爬技能` / `触发上升气流`）。模型按"塞尔达原版游戏实况截图复刻"+ 动作动词，就能从 BOTW/TOTK 游戏内动画外观自然生成对的姿态。
  - 反例：把"滑翔伞起跳"写成"伞才半展开 60%、脚仍踩在木板上未离地、双膝微屈像即将起跳前的最后一帧"——这种描述反而让模型把字面意思塞进画面，生成"原地撑伞站地"等游戏内不合法的姿态。
  - 正例：写"林克张开滑翔伞从观景台跳下"即可。模型会按 BOTW/TOTK 游戏内滑翔动画生成"腾空 + 伞撑开"的合法姿态。
- **场景特征保留**：真实景区的建筑、地形、水体、植被、文化符号要保留并融入海拉鲁画面语言；现代旅游设施（闸机、护栏、售票窗口、缆车站、塑料标识牌等）做游戏化转译（→ 驿站、木牌、机关、马车索道、HUD 标签、远景剪影），不要写实出现。
- **人物全部 NPC 化**：游客、工作人员、排队人群、摊主、导游、路人都表现为塞尔达村民、驿站 NPC、旅行商人、研究员、守卫或冒险者 NPC——不要现代游客服装、写实路人、自创画风。
- **NPC 服装黑名单**：禁止现代中国民族服装（汉服、唐装、旗袍、藏袍、维吾尔族花帽、苗银、白族头饰、彝族披毡、纳西披星戴月、僧侣袈裟、道士青袍、农民工作服、清代官服、民国学生装、解放军军装、武术服、太极服、农妇围裙背带裙）。NPC 必须从下列海拉鲁人物池选一种：海利亚村民布衣、驿站老板/旅伴马甲、海利亚商人挑担、城堡守卫银甲、希卡族黑紫忍装+面巾、卓拉族鱼鳞银身、利特族鸟形羽翼、鼓隆族岩石巨身、格鲁德族沙漠白纱+尖耳、海拉鲁研究员白大褂。

## 2. HUD 与精力条规则

HUD 是真实游戏实况截图里**自然出现**的界面状态，不是堆满屏幕的信息图，也不是精确像素规格。

### 基础 HUD（按画面叙事自然出现 1-3 种，不要堆满）

- 小地图
- 心心血量
- 武器或道具图标
- 互动按键提示
- 任务标记
- 天气/时间显示
- 区域发现地名提示
- NPC 对话框

### 耐力轮/精力条（条件性 HUD）

只在林克**正在消耗体力**的动作里出现：

- 跑步冲刺、攀爬、游泳、滑翔、持弓蓄力、骑马加速 → **加耐力轮**
- 站立、慢走、观察风景、对话、拍照、解谜等待、普通持剑戒备、看远处目标 → **不要加耐力轮**

写 prompt 时**先想清楚林克这张图在做什么**，再决定要不要写耐力轮。没有体力消耗动作就明确写"HUD 有小地图、心心血量和互动提示，但不显示耐力轮/精力条"。多张图任务里不要每张都默认带耐力轮——让 HUD 随玩法行为变化。

### UI 不要钉死屏幕方位

**不要在 prompt 里写"左上角小地图""右下角按键提示""屏幕中上区域发现""屏幕右上角图鉴登录"等屏幕方位词**。塞尔达游戏 UI 本来就有标准布局，生图模型按"原版实况截图复刻"心锚自然处理就行——写死方位反而让模型瞄错布局。

### HUD 字段写得越简越好（重要）

**不要在 prompt 里写死 HUD 的具体内容、形状、位置、排列**——HUD 是 image_gen 该按"BOTW/TOTK 原版游戏实况"心锚自然处理的元素，不是脚本字段。写得越细，模型越会把字面意思塞进画面，反而破坏游戏实况的自然感。

具体不要写：

- ❌ HUD 元素的**形状描述**（例：`完美正圆形小地图`、`圆角矩形小地图`、`外圈带细密放射刻度`、`圆周外圈刻度环`、`顶部 N 方向指针`、`内部地形等高线`、`红色玩家位置箭头`、`一排红色实心爱心连续无间断`、`绿色圆环耐力轮`、`圆形 A 按键`）
- ❌ HUD 元素的**位置/排列描述**（例：`左上角心心`、`右上角小地图`、`小地图正下方紧贴一列`、`贴右上 HUD 区右边缘`、`屏幕底部中央`、`屏幕中上偏上`、`垂直紧密排列`、`间距小`）
- ❌ HUD 元素的**数量描述**（例：`10 颗满血红心`、`3 个原版方形图标垂直排列`、`单按键`、`不要堆四按键`）—— 攻略图心心硬指标 10 颗除外，但只在 `UI 元素` 字段写一句话即可，不写形状细节
- ❌ 具体武器图标内容（例：`武器图标（蓝色波克布林角弓）`、`道具栏：火焰之剑+王者之盾`）
- ❌ 耐力轮状态（例：`耐力轮显示中`、`精力条剩 1/3`、`紧贴林克头顶`）
- ❌ 小地图上具体内容（例：`小地图右上有任务标记`、`小地图内部蓝色湖湾轮廓`）
- ❌ 按键提示的**按键码和样式**（例：`A 调查`、`X 滑翔`、`圆形里写 A`）—— 改为只列允许出现的中文动词清单

应该写：

- ✅ 一句方向性说明：`HUD：按 BOTW/TOTK 原版游戏实况自然出现，不堆满`
- ✅ 在 `林克本图动作` 字段说明本图是否消耗体力，由 image_gen 决定要不要画耐力轮（不要在 HUD 字段额外指挥）
- ✅ 如本图玩法关键到必须出现某类元素（如「祈祷增加心心」依赖心心动画），用一句简短叙事提一下：`HUD 体现心心增加的动画反馈`
- ✅ 区域发现/对话框的**中文文案内容**还是要写（这是 prompt 里 image_gen 拿不到的信息，例：`地名 UI 文字（区域发现）：顶部「发现！」 / 中间「喀纳斯·观鱼台」 / 底部「新疆·阿勒泰」；按原版区域发现三段式呈现，底部上级区域右对齐、只在文字左侧带短横线、淡色发光、无矩形边框、无深色底纹`、`NPC 对话框「商人：要点辣的吗？」`），但不要写它在屏幕的哪一侧
- ✅ 允许出现的中文按键动词清单（例：`允许的按键动词：调查 / 拾取 / 攀爬 / 滑翔`），由模型自己选 1-2 个落到画面

**核心准则**：HUD 字段值越短越好，理想长度 ≤ 2 行。出现"形状/位置/数量/排列/像素描述"几乎一定是错的——这种细节交给 image_gen 按原版游戏实况心锚自处理。如果 HUD 缺失或位置错，那是 **image_gen 的复现失败**，交给用户在出图后人工审，**不是靠把 prompt 写得更细来强压**。

理由：写死具体武器名、耐力轮状态、小地图形状、按键提示样式会让 image_gen 把这些信息按字面塞进 HUD，**反而生成出贴纸式 / 信息图式 / 现代 App 风的伪 HUD**；而把决策交回模型反而能拿到更像真实游戏截图的 HUD。

## 3. UI 文字中文化规则

画面里的可读文字要像中文本地化版本的游戏 UI，**不要出现英文界面词**。

### 必须显示中文的位置

按键提示、互动提示、攻击/跳跃/拾取/对话/调查提示、任务提示、区域发现提示、NPC 对话框、地图标记、图鉴界面、路牌、石碑文字。

### 中文按键动词清单

```
攻击、对话、调查、拾取、打开、攀爬、滑翔、跳跃、潜行、蓄力、瞄准、骑乘、丢出、烹饪、装备
```

**严禁出现的英文 UI 词**：Attack、Talk、Examine、Pick up、Open、Climb、Glide、Jump、Sneak、Aim、Ride、Cook、Equip 等。

### 文字无法清晰时

优先**减少**画面内文字，只保留极简中文 UI 词或编号；不要乱码、伪英文、外挂字幕或旅游说明文字。

## 4. 区域发现地名 UI 样式

场景图和攻略关键点展示图默认都要写一行 `地名 UI 文字（区域发现）`，用塞尔达原版"发现！"区域发现 UI 的**三段式版式**。只有攻略地图、纯地图界面或用户明确要求不显示地名时才省略。

- **顶部行**：小字"发现！"（带感叹号），居中，两侧各配一条短横线装饰
- **中间行**：大字主地名，居中
- **底部行**：小字副标题（主地名的上级真实区域），**右对齐**（贴近中间行右端的下方，不是居中），**只在文字左侧**带一条短横线装饰，右侧不带
- 整体淡色发光、淡入淡出感
- **没有矩形边框、没有深色底纹、没有花纹/纹章/图腾装饰**（除上面定义的短横线外不加其他装饰）
- **字号约束**：中间行主地名文字宽度不超过画面横向 1/2，高度不超过画面纵向 8%；不是旅游海报标题级别的占满屏大字

**所有 UI 地名文字必须是真实中文地名，严禁塞尔达原版地名**：区域发现 UI（三段全部）、路牌、石碑、地图标记、NPC 对话框里**所有可读地名文字**，都只能写本图对应的真实中国景区/省市/景点/区域名（例：中间行「阿里·扎达土林与古格王朝遗址」、底部副标题「西藏·阿里地区」；中间行「拉萨·布达拉宫」、底部副标题「西藏·拉萨」；中间行「张家界」、底部副标题「湖南·张家界」）。**禁止**出现「海拉鲁」「格鲁德高地」「格鲁德沙漠」「海利亚村」「卡卡里科村」「鼓隆城」「卓拉领地」「双子山」「海拉鲁城堡」等任何塞尔达原版地名/区域名。原因：UI 里用塞尔达原版地名会让画面变成"塞尔达截图"而不是"真实景区的塞尔达化呈现"，丢掉地点叙事。prompt 里如果用了"海拉鲁地图区块：XXX"这类字段名作为锚点，**字段名是写给模型理解画风用的，UI 文字必须落到 XXX（真实地名），不能把"海拉鲁"或其他原版地名搬进 UI**。

**固定 prompt 写法**（每张场景图 / 攻略关键点展示图直接替换地名使用）：

```text
地名 UI 文字（区域发现）：顶部「发现！」 / 中间「<真实主地名>」 / 底部「<真实上级区域名>」；按原版区域发现三段式呈现，底部上级区域右对齐、只在文字左侧带短横线、淡色发光、无矩形边框、无深色底纹
```

也可做成地图标记、路牌、石碑、NPC 对话框、任务提示或图鉴/地图界面文字。

**禁止做成**：旅游海报标题、宣传横幅、说明字幕、右上角固定标签、二维码、emoji。场景图和攻略关键点展示图默认加入三段式地名 UI；只有攻略地图、纯地图界面或用户明确要求不显示地名时才省略。

## 5. 原版游戏元素池

按地点气质和画面构图，从下列六大类挑选组合：

### 主角与重要角色
林克、塞尔达、公主、贤者、希卡族、利特族、卓拉族、鼓隆族、格鲁德族、海利亚村民、旅行商人、驿站老板、巡逻守卫

### 敌人与生物
波克布林（红/蓝/黑）、蜥蜴怪、莫力布林、人马、蝙蝠怪、丘丘、骷髅怪、岩石巨人、方块魔像、古代守护者、守护者残骸、飞行魔物

### 探索建筑
原版神庙、试炼神庙入口、高塔、瞭望塔、鸟望台、古代石门、地下洞口、漂浮岛、天空遗迹、石环机关、传送台、试炼入口、女神像、驿站

**神庙入口标准外观（必须写进 prompt，不要让模型理解成「发蓝光的小门 / 牌坊」）**：
- 一整面**橙色发光的「希卡眼睛」徽记**（一只竖瞳大眼 + 三根下垂泪滴）刻在入口正中
- 外圈一组**缓慢旋转的浅橙石环 + 浅橙光晕**
- 入口本体是**深灰石质方门或拱门**，门口透出橙光（未通关）或蓝光（已通关）
- 嵌进山体/建筑时保留石环和眼睛徽记，**不要画成中式牌坊/佛龛/木门 / 发蓝光的发光门洞**

**女神像标准外观**：石质坐像 / 立像，头戴海拉鲁三角额印，双手合掌或胸前交叠，**淡白/淡蓝光晕**，造型偏几何柔和——不要画成中式金身佛像、不要画成多臂观音、不要画成具象人体写实雕塑。

**鸟望台 / 高塔标准外观**：木质或左纳乌石质三层结构 + 顶层鸟形雕饰平台（鸟望台）或玛雅式高塔金字塔顶（古代高塔），**禁止**画成纯中式飞檐塔楼当鸟望台。

### 互动道具
原版宝箱、篝火、料理锅、火把、木桶、炸药桶、矿石、武器架、盾牌、弓箭、滑翔伞、马匹、木筏、风扇、火箭、热气球、左纳乌装置、古代机关控制台

### 玩法与 HUD
小地图、心心血量、按动作需要出现的耐力轮/精力条、道具切换图标、互动按键提示、准星、任务标记、天气时间显示、敌人警戒符号、拍照图鉴框、望远镜标记、区域发现提示、地图标记、NPC 对话框

### 环境玩法
敌人营地、巡逻路线、可攀爬岩壁、破碎吊桥、风场、瀑布攀登点、湖中隐藏宝箱、山洞入口、解谜石碑、机关门、可点燃火盆、可破坏岩墙

## 6. 元素差异化原则

不要把所有元素排成清单式摆拍；要像游戏实况中**自然出现**的探索、战斗、解谜和村落活动。

- **单张图**：至少 5-10 个原版游戏元素。建议组合 = 1 个林克（或塞尔达）+ 1-3 个敌人/生物 + 1 个神庙/高塔/鸟望台 + 2-4 个互动道具或环境玩法 + 适量 HUD。
- **多张图**：每张图元素组合必须不同；相邻两张不要重复超过 3 个核心元素。可以都出现林克，但姿态、距离、装备、行动目标要不同。
- **角色比例**：林克通常是前景或中景的可操作角色，景区地标仍然是大环境主体。除非用户明确要求角色特写，不要做宣传海报式大头像。
- **绑定真实景点特征**：宝箱藏在桥下、神庙嵌入山体、高塔立在山脊、敌人营地占据古道或湖边空地、滑翔路线穿过真实地貌、马匹拴在驿站门口——元素要和景点的真实建筑/地形产生关系，不是悬浮在场景上的贴纸。

## 7. 必须避免清单

- 不要现代旅游海报、扁平信息图、卫星地图、纯概念设定图、宣传 Logo、水印
- 不要把所有 UI 元素堆满；要像一张正在游玩的游戏截图
- 不要让 UI、特效、战斗爆炸盖过景区主体
- 不要现代 App 风的导航栏 / badge 数字 / 弹窗 × / emoji / 二维码 / 旅游海报标题大字
- 不要写实摄影、影视剧照、奇幻 CG、动漫插画、自创画风
- 不要写实游客服装、现代制服工作人员、真实路人
- 除非用户明确要求标题画面，否则不要生成游戏标题页、封面、商标页
